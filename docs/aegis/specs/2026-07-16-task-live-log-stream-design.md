# 任务实时日志流与独立日志页面设计

- 状态：已批准，待用户审阅书面规格
- 日期：2026-07-16
- 范围：后台任务日志采集、持久化、实时传输、受限 Docker 日志读取和独立任务日志页面
- 依赖：`2026-07-16-recent-job-logs-retention-design.md`

## 1. 目标

管理员应能查看尽量所有后台任务的完整执行输出，而不只看到结构化阶段事件。首版覆盖 SteamCMD 安装与更新、插件包部署、私有文件应用、GitHub Release 获取、计划任务触发及其他通过任务管理器执行的操作。

独立任务日志页面应支持实时追加、自动跟随底部、用户上滚后暂停跟随、手动恢复跟随、关键词搜索、来源和级别筛选、复制以及下载。任务详情和最近任务列表提供进入日志页面的命令。

日志必须跨 Panel 重启保留，并跟随任务记录生命周期。每个已结束任务最多保留最后 10 MiB 日志；运行中任务不截断，进入终态后再收敛到最后 10 MiB，并明确记录早期内容已被截断。任务记录被清理时同步删除对应日志。

## 2. 非目标

- 不提供跨任务全文索引或跨任务聚合搜索。
- 不把日志作为审计记录，也不改变审计事件语义。
- 不提供用户可配置的日志大小、压缩算法或独立时间保留期限。
- 不保证所有第三方命令输出都有可靠的日志级别；无法识别时使用 `output`。
- 不允许通过 API 读取任意宿主机文件、任意容器日志或非任务容器输出。
- 不改变任务串行化、取消、重试、调度和成功任务数量保留规则。

## 3. 方案选择

采用追加式文件持久化、轻量元数据和 SSE 实时传输。

每个任务拥有一个追加日志文件，位于数据根下的 Panel 私有目录，例如 `panel/job-logs/<job-id>.log`。SQLite 只保存日志元数据和现有结构化 `job_events`，避免高频输出放大 WAL、锁竞争和数据库备份体积。

未采用 SQLite 分块正文存储，因为 SteamCMD 等任务可产生大量连续输出，频繁事务会与任务快照和设置写入竞争。未采用纯内存广播，因为 Panel 重启会丢失诊断证据，也无法可靠断线续传。

结构化事件与原始日志是两个互补来源：`job_events` 继续表达排队、开始、阶段、成功和失败；日志文件保存外部命令与内部任务执行器的实际输出。页面可以合并展示，但两者不互相替代。

## 4. 日志记录格式

日志文件使用 UTF-8 JSON Lines。每行至少包含：

- 单调递增的任务内序号 `seq`。
- UTC 时间 `timestamp`。
- 来源 `source`，例如 `task`、`steamcmd`、`docker`、`package`、`private-files`、`release`。
- 级别 `level`，允许 `output`、`info`、`warn`、`error`。
- 文本 `message`。

写入器按完整记录追加，不允许多个调用方直接写文件。无法解码的外部字节流使用 UTF-8 replacement character 转换；ANSI 控制序列在持久化前移除，普通换行拆分为独立记录。空行可以保留，但连续高频进度覆盖行应以合理频率合并，避免 SteamCMD 百分比刷新造成无意义膨胀。

结构化任务事件也写入日志来源 `task`，但数据库 `job_events` 仍是任务详情时间线的规范所有者。日志写入失败不得把已经成功的业务操作改判失败；任务终态应附加一条日志子系统错误，并在 Panel 服务日志中记录。

## 5. 持久化与恢复

新增日志元数据，至少包含任务 ID、当前字节数、最后序号、是否已截断、创建时间和更新时间。元数据可以存放在独立 SQLite 表中，也可以由日志管理器在启动时从文件恢复；实现计划应选择一个规范所有者，不能同时维护两个可漂移的大小来源。

日志管理器负责：

- 为任务创建日志并追加记录。
- 对同一任务串行写入并在广播前完成持久化。
- 按序号或字节偏移读取历史片段。
- 任务进入终态后保留最后 10 MiB，并在新文件首行写入 `task/warn` 截断标记。
- 任务记录被删除后删除日志和元数据。
- Panel 启动时清理没有任务记录的孤儿日志，并恢复现有任务的最后序号。

运行中日志不受 10 MiB 上限约束，以避免正在执行时删除管理员尚未读取的输出。磁盘耗尽时写入器应停止追加、记录服务端诊断并通过任务日志状态暴露“日志采集已停止”，但不得终止外部任务。

Panel 重启恢复 `pending` 或 `running` 任务为 `interrupted` 时，先追加中断记录，再执行 10 MiB 终态截断。

## 6. 输出采集

所有任务通过统一 `JobLogger` 接口记录内部阶段、警告和错误。现有 `Reporter.Progress` 继续更新结构化事件，并同时向日志写入可读记录。

SteamCMD 维护容器是首要外部输出来源。Docker 客户端在容器启动后并行跟随 stdout/stderr，在容器退出后读取到 EOF，再等待退出码、写入最终状态并删除容器。日志跟随失败应保留任务执行，最终错误同时包含 SteamCMD 退出码和日志采集故障摘要。

Docker socket proxy 新增受限日志接口，只允许读取同时满足以下条件的容器：

- `io.l4d2-panel.managed=true`。
- 角色为 `maintenance`。
- 请求由现有 Unix socket 进入。

代理只开放容器日志读取所需的固定查询参数，拒绝任意容器、任意路径和不受限 follow 设置。游戏控制台继续由 supervisor 通道管理，不复用任务日志接口。

插件部署、私有文件应用和 Release 获取在其规范执行器边界写入来源化日志。首版不为没有外部输出的纯数据库操作制造伪详细日志，仍至少记录开始、关键阶段和终态。

## 7. HTTP 与 SSE 契约

新增经过现有管理员会话认证的接口：

- `GET /api/jobs/{id}/logs`：按 `after_seq`、`limit`、来源和级别读取历史记录。
- `GET /api/jobs/{id}/logs/stream`：SSE 实时流，支持 `Last-Event-ID` 或 `after_seq` 续传。
- `GET /api/jobs/{id}/logs/download`：下载该任务当前完整保留日志。

历史接口响应包含记录、下一序号、是否还有更早数据、是否已截断和任务是否仍在运行。默认从最新一页读取，页面向上滚动时加载更早记录。单次响应必须有记录数和响应体大小上限。

SSE 事件 ID 使用日志序号。建立连接时先补发客户端缺失的已持久化记录，再订阅新记录，避免“读历史”和“开始监听”之间丢行。慢客户端使用有界队列；队列溢出时断开连接，让浏览器通过最后事件 ID 重连，而不是阻塞任务写入。

下载使用由任务 ID 生成的固定文件名，不能接受路径参数。任务不存在返回 404；任务存在但尚无日志返回空文件或空记录集合；日志已被任务清理删除时与任务不存在保持一致。

## 8. 前端页面

新增独立任务日志视图，并从最近任务列表和任务详情进入。页面顶部显示任务类型、短 ID、实例、状态、阶段、开始/结束时间、执行时长和失败摘要。

日志工具栏包含：

- 搜索输入，匹配当前已加载记录并可继续向历史页搜索。
- 来源多选菜单和级别多选菜单。
- 自动跟随开关或明确的恢复跟随命令。
- 复制当前筛选结果。
- 下载完整保留日志。

日志区域使用稳定高度的等宽文本视图。初次打开默认定位底部并自动跟随。用户向上滚动离开底部后暂停跟随，仍在内存接收新记录并显示未读数量；点击恢复后滚到底部并继续跟随。筛选和搜索不应破坏接收流或改变服务器任务。

断线时保留已加载内容并显示重连状态。历史加载失败只影响向上加载，实时流失败只影响新增内容，两者分别可重试。截断任务在最早可见位置显示“早期日志已因 10 MiB 上限截断”。最长单行必须换行或水平滚动，不能撑破页面布局。

复制只包含当前筛选后的可见文本；下载始终返回服务器保留的完整日志，不受前端筛选影响。运行中任务下载得到请求时刻的稳定快照。

## 9. 脱敏与安全

日志进入持久化前执行集中脱敏。至少覆盖：

- Steam 用户密码和 Panel 管理员密码。
- `Authorization`、`Cookie`、会话令牌和常见 API token 格式。
- 已加载到 secrets 服务中的 Steam 与 GitHub 凭据。
- 环境变量形式的 `*_PASSWORD`、`*_TOKEN`、`*_SECRET`。

脱敏器只在一个规范边界执行，调用方不能选择跳过。原始秘密不得先写文件再异步清理。对于无法可靠解析的命令行，命令记录默认只写允许列表参数或经过脱敏的展示形式。

所有日志接口继续使用管理员会话和现有 CSRF/同源边界。Socket proxy 的容器标签校验必须发生在代理内部，不能信任 Panel 传来的角色声明。

## 10. 清理与兼容边界

现有成功任务数量保留设置仍决定哪些任务记录被删除。任务记录删除是日志删除的唯一正常触发条件；失败、中断和运行中任务继续遵守现有任务记录保留规则。

每个终态任务独立应用 10 MiB 日志上限，不是所有任务共享总额度。调高或调低成功任务记录数量不会改变仍保留任务的 10 MiB 上限。

现有 `/api/jobs`、`/api/jobs/{id}`、`job_events`、任务 SSE 摘要和最近任务页面保持兼容。新增日志接口和页面是加法式扩展。旧任务没有日志文件时仍可查看原有结构化事件，并显示“此任务执行时尚未启用完整日志采集”。

本规格取代旧规格中“不采集外部完整 stdout/stderr、不提供全文搜索和下载”的非目标，但不改变旧规格其他已批准内容。

## 11. 测试策略

- 日志管理器：并发追加稳定顺序、UTF-8、ANSI 清理、进度行处理、重启恢复、孤儿清理、运行中不截断、终态 10 MiB 截断和任务删除联动。
- 脱敏：密码、token、请求头、环境变量、跨块秘密及误匹配边界。
- Docker 客户端：启动后跟随日志、读取到 EOF、退出码、采集失败不覆盖业务结果、删除顺序和绝对 SteamCMD 入口。
- Socket proxy：只允许受管 maintenance 容器日志，拒绝游戏容器、未标记容器、非法参数和越权 ID。
- Job Manager：结构化事件同步写日志、终态截断、恢复中断记录和日志故障降级。
- HTTP：历史分页、默认最新页、筛选、SSE 无缝补发、Last-Event-ID、慢客户端重连、下载、404、空日志和截断元数据。
- React：默认跟随、上滚暂停、未读计数、恢复、搜索、来源/级别筛选、复制、下载、历史加载、断线重连和截断提示。
- E2E：启动可控输出任务，打开独立日志页，观察实时追加，上滚后不跳动，恢复跟随，筛选、复制、下载，并验证失败任务显示 SteamCMD 原始错误。
- 回归：全部 Go 测试、`go vet`、前端单测、生产构建和桌面/移动端 Playwright。

## 12. 实施边界与分阶段

实施计划应按以下依赖顺序拆分原子任务：

1. 日志记录格式、文件存储、截断、恢复和脱敏。
2. Job Manager 接入内部结构化输出。
3. Socket proxy 与 Docker 客户端受限日志跟随。
4. 各任务执行器接入来源化日志。
5. 历史、SSE 和下载 HTTP 契约。
6. 独立 React 日志页面及入口。
7. 真实 SteamCMD 失败链路和浏览器 E2E 验证。

SteamCMD 当前入口覆盖修复属于前置缺陷修复，可独立提交；完整日志功能不得依赖保留调试容器或绕过受限 socket proxy。

## 13. 设计输入与影响

### TaskIntentDraft

让管理员对尽量所有后台任务查看可持久、可实时跟随的完整日志；提供独立日志页面、搜索、来源/级别筛选、复制和下载。运行中日志不截断，每个终态任务最多保留最后 10 MiB，日志跟随任务记录删除。

### BaselineReadSetHint

现有任务系统已持久化 `jobs` 快照和 `job_events` 结构化事件，并通过任务摘要 SSE 更新最近任务页面；旧规格明确排除了外部 stdout/stderr。Docker 访问通过受限 Unix socket proxy，SteamCMD maintenance 容器执行后会被删除，因此完整日志必须在删除前持续采集。

### ImpactStatementDraft

变更影响任务管理器、SQLite 元数据、数据根文件布局、Docker 客户端、socket proxy 策略、SteamCMD 和其他任务执行器、HTTP/SSE 契约、最近任务入口及新 React 页面。兼容边界是保留现有任务状态、事件、保留数量和执行语义；新增日志是加法式诊断能力，日志故障不得反向改变业务任务结果。
