# 计划任务管理与任务说明设计

日期：2026-07-16

## 1. 目标

修复“已经创建的计划任务无法编辑或删除”的管理缺口，并为所有计划任务类型提供可从页面打开的详细中文说明。

编辑采用受限语义：管理员只能修改 Cron 表达式、在线玩家策略和启用状态。任务 ID、任务类型、目标实例、时区以及任务自身参数保持不变。删除采用二次确认，并明确删除计划不会取消已经排队或正在执行的 Job。

## 2. 根因与方案选择

后端已经提供以下能力：

- `POST /api/schedules` 在请求带有既有任务 ID 时更新该记录，并重新注册或移除 Cron 条目。
- `DELETE /api/schedules/{id}` 删除数据库记录和对应 Cron 条目。

当前 React 页面只提供新建表单；任务列表没有编辑或删除入口。列表归一化还丢弃了 `instance_id`、`online_policy`、`payload`、`last_run` 和 `next_run`，因此前端不能安全回传既有任务。

采用前端复用现有 API 的方案，不新增第二套更新所有者：

1. 前端保留完整的 `ScheduledTask` 数据契约。
2. 编辑表单以完整任务为基线，只开放已确认的三个字段。
3. 保存时回传完整记录，使后端现有 upsert 成为唯一更新路径。
4. 删除使用现有 DELETE 路由和通用确认对话框。
5. 任务说明由一个集中式前端目录定义，选择器、列表名称和说明弹窗共用同一来源。

不新增 `PUT` 路由，也不建立独立计划详情页。现有 API 已能表达所需行为，增加平行接口只会形成重复所有权。

## 3. 编辑行为

每条计划任务提供带工具提示和可访问名称的编辑图标按钮。点击后，左侧表单进入编辑模式并显示任务的中文名称和目标；任务类型、目标实例、时区、插件包、GitHub 源、保留天数等身份或任务参数只读。

可编辑字段如下：

- `cron`：五段 Cron 表达式。保存后以后端当前时间重新计算 `next_run`。
- `online_policy`：仅对存在目标实例并会检查玩家的任务开放；可选“跳过”“等待”“强制执行”。
- `enabled`：停用时保留数据库记录但从 Cron 调度器移除；重新启用时重新计算下一次执行时间并注册 Cron 条目。

保存请求必须保留下列字段：`id`、`instance_id`、`type`、`timezone`、`payload` 和 `last_run`。`next_run` 可随请求回传，但调度服务以新 Cron 重新计算的值为准。保存成功后退出编辑模式、刷新列表并显示成功状态；失败时保持编辑内容并在表单内显示错误。

“取消编辑”恢复新建模式，不发送请求。编辑中的任务若被刷新结果删除，页面退出编辑模式并提示该记录已不存在。

## 4. 删除行为

每条任务提供删除图标按钮。点击后打开确认对话框，内容包括任务中文名称、Cron、目标名称，并明确：

> 删除后不会再按此计划触发新任务；已经排队或正在执行的任务不会被取消。

确认后调用 `DELETE /api/schedules/{id}`。成功时刷新列表；若当前正在编辑该任务，同时退出编辑模式。失败时保留任务和编辑状态，并显示服务端错误。取消确认时不发送请求。

删除不删除 Job 历史、备份、已下载插件包、GitHub 源或实例数据。

## 5. 列表信息

计划列表使用任务卡片而不是仅显示内部类型字符串。每项至少显示：

- 中文任务名称和启用/停用状态。
- 目标游戏实例；全局任务显示“全局”。
- Cron 和创建时确定的时区。
- 在线玩家策略；不适用时显示“不检查玩家”。
- 上次执行时间和下次执行时间；无记录时显示“尚未执行”或“未安排”。
- 编辑和删除图标按钮。

列表不显示原始 JSON `payload`。插件包、GitHub 源和清理保留天数可解析为可读摘要；引用对象已经删除时显示“引用已失效”，但仍允许停用或删除该计划。

## 6. 任务说明入口

计划任务页面标题区增加“任务说明”按钮。点击后打开可滚动对话框，完整列出八种任务。弹窗支持 Escape 关闭、键盘焦点约束和移动端滚动；关闭不改变当前表单。

每种说明统一包含：执行目标、实际步骤、是否可能中断实例、在线玩家策略生效点、所需参数、成功产物和主要注意事项。新建表单仍使用简洁中文名称，不把长说明塞进下拉选项。

## 7. 八种任务的精确语义

### 7.1 游戏更新 (`game_update`)

- **目标：** 一个游戏实例。
- **步骤：** 先应用在线玩家策略；允许执行后读取实例状态，必要时停止实例，通过维护流程运行 SteamCMD 更新或校验游戏文件，重新应用私有文件覆盖，然后按原期望状态决定是否启动实例。
- **中断：** 运行中或启动中的实例会停止；原本期望运行的实例在更新和健康检查成功后恢复运行，原本停止的实例保持停止。
- **参数：** 无额外任务参数。
- **注意：** 更新时间取决于 Steam 下载；失败会形成 `scheduled_game_update` Job 错误，实例可能进入故障状态并需要检查日志。

### 7.2 插件热更新 (`package_hot`)

- **目标：** 一个游戏实例和一个已经存入内容仓库的插件包。
- **步骤：** 先应用在线玩家策略，再读取指定插件包；仅部署热更新允许的配置、脚本和插件内容，最后重新应用私有覆盖并记录该包为已应用版本。
- **中断：** 不主动停止或重启实例。
- **参数：** `payload.package_id` 必须引用存在且通过热更新兼容检查的插件包。
- **注意：** 不适合需要替换游戏二进制或必须重启才能生效的内容；包不兼容时任务失败，不自动降级为完整更新。

### 7.3 插件完整更新 (`package_full`)

- **目标：** 一个游戏实例和一个已经存入内容仓库的插件包。
- **步骤：** 先应用在线玩家策略；活动实例先停止，事务化部署插件包并重新应用私有覆盖，再按更新前状态启动并通过健康检查；成功后记录已应用版本。
- **中断：** 活动实例会停止并可能断开玩家；原本停止的实例保持停止。
- **参数：** `payload.package_id` 必须引用存在的插件包。
- **注意：** 启动或健康检查失败时部署事务回滚，并尝试恢复更新前内容和运行状态。

### 7.4 仅同步 GitHub 源 (`release_check`)

- **目标：** 一个 GitHub Release 源，不绑定游戏实例。
- **步骤：** 使用源的仓库和资源匹配规则检查最新 Release；需要时使用已配置的 GitHub Token；找到新的匹配资源后下载、校验并登记为内容仓库中的插件包。
- **中断：** 不停止、不更新任何游戏实例，也不自动应用下载的包。
- **参数：** `payload.source_id` 必须引用存在的 GitHub 源。
- **注意：** 这是全局同步任务，不检查在线玩家；没有新版本时正常结束。

### 7.5 GitHub Release 热更新 (`release_hot`)

- **目标：** 一个游戏实例和一个 GitHub Release 源。
- **步骤：** 先检查并下载最新匹配 Release；只有发现新版本时才应用在线玩家策略，然后以热更新方式部署新包并记录已应用版本。
- **中断：** 不主动停止或重启实例。
- **参数：** `payload.source_id` 必须引用存在的 GitHub 源。
- **注意：** Release 下载发生在玩家检查之前；没有新版本时不部署。新包不满足热更新规则时任务失败，不自动改为完整更新。

### 7.6 GitHub Release 完整更新 (`release_full`)

- **目标：** 一个游戏实例和一个 GitHub Release 源。
- **步骤：** 先检查并下载最新匹配 Release；只有发现新版本时才应用在线玩家策略，然后执行完整插件更新：停止活动实例、事务化部署、重新应用私有覆盖、按原状态启动并检查健康。
- **中断：** 活动实例会停止并可能断开玩家。
- **参数：** `payload.source_id` 必须引用存在的 GitHub 源。
- **注意：** Release 下载发生在玩家检查之前；没有新版本时正常结束。部署或重启失败时执行回滚。

### 7.7 备份 (`backup`)

- **目标：** 一个游戏实例。
- **步骤：** 先应用在线玩家策略，然后将当前私有文件工作区和已应用插件清单写入实例 `backups/` 下的时间戳 `.tar.gz`；归档完成并同步后才原子发布。
- **中断：** 不停止或重启实例。
- **参数：** 无额外任务参数。
- **注意：** 备份不包含完整游戏安装、容器镜像、SQLite 数据库或历史备份；遇到符号链接或写盘失败时不会发布不完整归档。

### 7.8 清理 (`cleanup`)

- **目标：** Panel 管理的数据根目录，不绑定单个实例。
- **步骤：** 删除保留期之前的实例 `backups/` 文件，以及包上传目录中遗留的 `.part` 和 `.upload` 文件；其他实例文件、当前私有工作区、内容包和数据库不在清理范围内。
- **中断：** 不停止或重启实例，也不检查在线玩家。
- **参数：** `payload.retention_days` 表示保留天数；缺失或小于 1 时按 30 天执行。
- **注意：** 清理按文件修改时间判断且不可撤销；已经删除的备份不会进入回收站。

## 8. 新建表单的任务参数完整性

为保证说明与实际可创建任务一致，新建模式补齐任务专属字段：

- `package_hot` 和 `package_full` 必须选择插件包，并写入 `payload.package_id`；热更新只提供 `hot_compatible=true` 的包。
- 三种 Release 任务必须选择 GitHub 源，并写入 `payload.source_id`。
- `cleanup` 提供保留天数输入，默认 30，并写入 `payload.retention_days`。
- 其他任务写入空 JSON 对象。

编辑模式不允许改变这些参数，只显示其可读摘要。这修复现有插件计划被保存为 `{}` 后必然找不到插件包的问题，但不改变调度器或执行器契约。

## 9. 错误处理与并发

编辑、删除和列表刷新分别维护明确的进行中状态，避免重复提交。异步完成时以任务 ID 校验当前编辑对象，防止较旧请求覆盖新选择。

服务端错误沿用现有 `errorMessage` 和页内错误区域。引用已经失效的插件包、GitHub 源或实例不会让整个页面崩溃；对应计划仍可停用或删除。编辑期间不尝试取消已经触发的 Job。

## 10. 兼容边界与退役路径

保持不变：

- `ScheduledTask` 数据库结构和 JSON 字段。
- `POST /api/schedules`、`GET /api/schedules`、`DELETE /api/schedules/{id}` 与手动运行接口。
- 调度服务对启用、停用、重新注册和下一次执行时间的现有语义。
- 自动化执行器的八种任务类型、在线策略和 Job 类型。

退役内容：任务列表只显示内部类型字符串并使用无动作箭头的旧渲染方式；计划任务类型名称的分散硬编码由集中目录替代。不会保留旧列表作为回退，也不会新增第二个更新 API。

## 11. 测试与验收

前端测试覆盖：

- 完整归一化已有任务数据。
- 编辑入口预填 Cron、策略和启用状态，并锁定类型、目标、时区和任务参数。
- 编辑只改变允许字段，保存请求保留完整任务身份和 `last_run`。
- 停用保存、取消编辑、保存失败保持输入。
- 删除确认、取消不请求、成功刷新、失败保留任务，以及删除当前编辑任务后退出编辑。
- 说明弹窗包含八种中文名称、执行步骤、中断影响、参数和注意事项，并支持关闭。
- 插件计划提交 `package_id`、Release 计划提交 `source_id`、清理计划提交 `retention_days`。
- 任务列表显示实例、策略、上次/下次执行时间和失效引用提示。

后端回归测试覆盖既有 ID 的保存会更新而非新增、重新计算下一次运行、停用移除 Cron 条目，以及 DELETE 后记录不再列出。若现有行为已完全满足，不修改后端生产代码。

浏览器主流程覆盖创建计划、编辑 Cron 并停用、重新打开页面确认持久化、删除并确认列表移除，以及打开任务说明弹窗检查全部类型。桌面与移动视口都不得出现按钮、说明或表单字段重叠。

验收标准是：管理员能安全编辑和删除任意已有计划；编辑不会改变任务类型、目标或参数；每种计划的实际行为、影响和风险都能从“任务说明”弹窗完整了解；新建的插件、Release 和清理任务携带执行器所需参数。

## 12. 设计输入与影响边界

### TaskIntentDraft

为计划任务补齐已有记录的编辑与删除操作，并为八种任务提供非常仔细、与实际执行器一致的中文说明。编辑范围按用户选择限制为 Cron、在线玩家策略和启用状态；任务说明使用独立弹窗。

### BaselineReadSetHint

设计以 `CONTEXT.md`、`internal/domain/models.go`、`internal/scheduler/service.go`、`internal/automation/dispatcher.go`、`internal/maintenance/manager.go`、`internal/updates/game.go`、`internal/updates/coordinator.go`、`internal/httpapi/server.go`、`web/src/app/App.tsx`、`web/src/styles/app.css` 及对应 Go/Vitest 测试为依据。

### ImpactStatementDraft

主要受影响所有者是 React 计划任务页面及其测试；后端仅补回归覆盖，除非测试证明现有契约有缺陷。必须保持的边界是单一调度更新路径、既有 JSON/SQLite 兼容、运行中 Job 不由计划删除操作取消，以及编辑不改变任务身份和执行参数。非目标是新增任务类型、Job 取消、批量编辑、任务运行历史删除和独立计划详情页。
