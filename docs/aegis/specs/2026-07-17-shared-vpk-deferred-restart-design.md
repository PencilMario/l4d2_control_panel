# 共享 VPK 空服延迟重启设计

## 目标

共享 VPK 成功发布后，为每个当前正在运行的游戏实例登记一次待处理重启。Panel 持续等待实例无人在线；确认空服后自动重启，使 supervisor 在 SRCDS 启动前把新 VPK 链接到 `left4dead2/addons/`。

多个 VPK 在等待期间发布时，每个实例仍只保留一个待重启事项，一次重启加载期间发布的全部共享 VPK。

## 术语

- **共享 VPK 待重启事项**：持久化记录，表示一个游戏实例尚需通过容器重启加载最新共享 VPK。它在等待期间不占用实例 Job 串行锁。
- **运行代次**：登记事项时记录的游戏容器 ID。容器 ID 变化表示该实例已由其他操作重建或换代。
- **异常玩家查询**：连续三次无法取得可信玩家快照；单次成功查询会将连续失败计数清零。

## 触发与实例选择

VPK 完成接口只有在新内容通过大小和 SHA-256 校验并原子发布至 `shared-vpk/` 后才触发协调器。相同 SHA-256 命中现有内容并返回 `duplicate=true` 时不触发。上传未完成、校验失败、目标文件名冲突或元数据写入失败均不触发。

协调器枚举全部游戏实例：

- 当前期望状态或实际状态为 `running`、`starting` 或 `installing`，且存在游戏容器的实例，登记待重启事项；
- `stopped`、`uninstalled` 及没有可用游戏容器的实例不登记，也不自动启动；
- `updating`、`rolling_back` 等维护状态由当时正在执行的实例操作负责形成新的运行代次，不额外登记；
- `faulted` 或 `orphaned` 不被上传流程强制启动。

登记采用每实例唯一约束。已有未完成事项时只更新其最新 VPK 发布标识和更新时间，保留原始登记容器 ID，不新增第二个等待者。

## 协调器与持久化

新增独立的共享 VPK 重启协调器，职责限定为事项登记、恢复、玩家轮询、状态转换和提交重启 Job。它不复用定时任务的 `OnlinePolicy`，也不在 HTTP 上传处理函数内执行长时间等待。

待重启事项持久化到 SQLite，至少包含：实例 ID、登记容器 ID、最新发布标识、状态、连续查询失败次数、创建时间、更新时间和下次检查时间。实例 ID 对未完成事项唯一。Panel 启动时加载未完成事项并恢复轮询。

状态为：

- `waiting`：等待玩家离开或累计查询失败；
- `queued`：已向 Job 管理器提交重启，等待实例串行锁；
- `retry`：重启失败，等待再次尝试；
- `completed`：已由本任务或其他运行代次变化满足；
- `cancelled`：实例被管理员停止或已不应自动启动。

完成和取消记录可按现有任务历史保留策略裁剪；未完成记录不得因 Panel 重启丢失。协调器启动、停止和定时器必须响应 context，避免进程退出时遗留 goroutine。

## 等待与重启流程

`waiting` 事项默认每 30 秒检查一次：

1. 重新读取实例和实际容器信息。
2. 若实例已停止或期望状态变为停止，标记 `cancelled`，不得重新启动。
3. 若当前容器 ID 与登记容器 ID 不同，说明其他操作已经产生新运行代次，标记 `completed`，不得再次重启。
4. 查询在线玩家。查询成功且人数大于零时保持等待，并把连续失败次数清零。
5. 查询成功且人数为零时提交重启 Job。
6. 查询失败时递增连续失败次数；前两次继续等待，第三次视为异常状态并提交重启 Job。

玩家人数使用现有 `players.Service.Online` 的合并快照，而不是只依赖 A2S Info 计数。查询失败包括 A2S、控制台状态或实例读取失败，但实例不存在和明确停止应走取消路径，而不是异常强制重启。

真正执行重启时，通过现有 Job 管理器创建类型为 `shared_vpk_restart` 的实例 Job。等待阶段不持有实例锁；Job 进入运行态后才与手动生命周期操作、更新、重建和私有文件应用串行。Job 开始执行前再次检查期望状态和容器 ID：已停止则取消，运行代次已变化则直接完成，否则调用现有 `Lifecycle.Restart`。

重启成功后事项标记 `completed`。重启失败时 Job 标记失败，事项进入 `retry`，按 30 秒固定间隔重新检查并重试；只要实例仍期望运行便不设置总超时。管理员停止实例会终止重试。

## 可观测性与接口

共享 VPK 完成响应保持现有 `item` 和 `duplicate` 字段，可增加已登记实例数量和登记失败诊断，但上传发布成功不能因个别实例登记失败而回滚已发布文件。登记数据库事务整体失败时返回明确诊断，并由服务端记录需要人工处理的实例。

每个实际重启使用现有 Job 列表、详情、事件和日志界面。Job 事件应包含：等待结束原因（空服或三次查询失败）、提交时的容器 ID、重启开始和最终结果。等待事项本身需要通过任务列表可见，展示“当前有 N 名玩家”“玩家查询失败 X/3”“等待实例操作”“重试重启”等状态，避免无限等待不可见。

本设计不新增配置页面。轮询周期、失败阈值和无限等待策略暂不开放用户配置。

## 并发与一致性

- 同一实例的事项登记必须使用数据库唯一约束或等价原子 upsert，防止并发上传产生多个事项。
- 事项从 `waiting/retry` 到 `queued` 必须为条件更新；只有一个协调器执行者可以提交 Job。
- 后续 VPK 发布更新已存在事项时不得覆盖原登记容器 ID，否则会误判已发生的运行代次。
- Job 提交失败时恢复为 `retry`，不能遗留永久 `queued`。
- Panel 在 Job 已提交但未完成时退出，现有 Job 恢复会将运行任务标记 interrupted；对应事项在下次启动时回到 `retry` 并重新核对实例状态。
- VPK 文件发布保持现有原子 rename、哈希去重和只读容器挂载语义。

## 测试与验证

- 新 VPK 发布仅为运行或启动中的实例登记事项；停止、未安装和异常实例不被启动。
- 重复哈希上传和失败上传不登记事项。
- 同实例连续或并发发布多个 VPK 始终只有一个未完成事项。
- 有玩家时持续等待；成功查询清零失败计数；零玩家触发一次重启。
- 连续三次查询失败触发重启，间隔成功不会误触发。
- 等待期间管理员停止实例会取消事项。
- 等待期间容器 ID 变化会完成事项且不二次重启。
- 等待阶段不阻塞其他实例 Job；执行重启时遵守实例串行锁。
- 重启失败进入无限重试，后续成功后完成；Panel 重启能恢复等待和重试事项。
- HTTP、任务列表和任务日志能表达登记结果、等待原因和重启结果。
- 回归验证共享 VPK 上传、下载、重命名、删除、资源清理、容器只读挂载和 supervisor 链接创建。

## 兼容边界与非目标

- 不重建 Docker 镜像或游戏容器；仅调用现有实例重启。
- 不改变全局 `shared-vpk` 只读挂载和 `addons` 符号链接机制。
- 不自动启动已停止、未安装、故障或孤立实例。
- 不改变现有计划任务 `skip/wait/force` 在线策略。
- 不为 VPK 重命名、删除或资源清理自动登记重启；本阶段只处理新 VPK 发布。
- 不增加强制超时、维护窗口或管理员可配置轮询参数。

## 设计输入

### TaskIntentDraft

共享 VPK 上传完成后，让所有当前运行实例在无人在线时自动重启；玩家查询连续三次失败时直接重启；等待无限期；每实例只保留一个事项；停止实例不处理。主要风险是误重启、重复重启、无限等待阻塞其他操作以及 Panel 重启后状态丢失。

### BaselineReadSetHint

- `internal/content/uploads.go` 与 `internal/httpapi/server.go`：VPK 发布和完成接口边界。
- `internal/jobs/manager.go`：持久 Job、实例串行锁和重启恢复语义。
- `internal/players/service.go`：可信在线玩家快照来源。
- `internal/lifecycle/service.go`：期望/实际状态与重启行为。
- `internal/automation/dispatcher.go`：现有在线等待实现及不适合直接复用的边界。
- `runtime/supervisor.py` 与 `internal/docker/lifecycle.go`：共享 VPK 挂载和启动时链接机制。

### ImpactStatementDraft

影响 HTTP 上传完成路径、内容协调服务、SQLite 存储、Job 可观测性、玩家查询和生命周期调用。保持 VPK 发布原子性、每实例操作串行、停止实例不被自动启动、内容优先级及容器只读挂载不变。前端仅复用或扩展现有任务可见性，不引入新的管理配置面。
