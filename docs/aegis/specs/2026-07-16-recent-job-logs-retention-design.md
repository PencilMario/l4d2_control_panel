# 最近任务日志、执行时间与成功记录保留设计

- 状态：已批准
- 日期：2026-07-16
- 范围：持久任务记录、任务详情 API、最近任务页面和系统设置

## 1. 目标

最近任务页面应让管理员点击任意任务查看其结构化执行日志。日志覆盖排队、开始执行、每次阶段或进度变化，以及成功、失败或中断结果；失败任务在展开区直接显示失败原因。

任务摘要应展示发起时间和实际执行用时。实际执行用时只计算开始执行到结束的时间，排队等待单独表达。

系统全局只保留最近若干个成功任务，默认值为 25，并允许管理员在系统设置中调整为 1 至 500。运行中、失败和中断任务不占成功任务额度。

## 2. 非目标

- 不采集或保存 SteamCMD、Docker 或其他外部命令的完整 stdout/stderr。
- 不提供全文日志搜索、下载、分页、批量删除或按实例分别配置保留数量。
- 不按时间自动删除失败或中断任务。
- 不改变任务串行化、调度、重试或取消语义。
- 不把任务事件并入审计事件；两者继续承担不同职责。

## 3. 方案选择

采用独立的追加式任务事件表。每次任务状态、阶段或进度变化都写入一条有序事件，同时更新现有任务快照。

未采用在 jobs 行内保存 JSON 数组，因为每次进度变化都要重写完整日志，写入成本随日志增长，并且不利于保证并发更新和事件顺序。未采用只展示现有任务快照，因为快照只保存最后一次阶段和消息，不能还原日志。

## 4. 数据模型与迁移

为 jobs 增加可空的 started_at 和 finished_at 字段。新任务进入实际执行时写入开始时间，进入 succeeded、failed 或 interrupted 时写入结束时间。

新增 job_events 表，至少包含：

- 单调递增的事件 ID。
- 所属任务 ID，并使用外键级联删除。
- 事件类型：queued、started、progress、succeeded、failed、interrupted，或仅由旧数据迁移生成的 snapshot。
- 事件发生时的阶段、进度、消息和 UTC 时间。

事件按事件 ID 正序返回；时间用于展示，事件 ID 用于稳定排序，避免相同时间戳造成顺序不确定。

新增普通系统设置表，以名称、值和更新时间保存非敏感设置。成功任务上限使用固定设置名，缺少记录时读取默认值 25。该设置不进入加密 secrets 表。

迁移保留现有任务。旧任务没有真实开始时间，因此仅以 created_at 作为兼容近似值，终态任务以 updated_at 回填结束时间；同时为每个旧任务生成一条合成事件，明确标记“升级前任务，仅保留最终快照，执行时间为估算值”，并带上已有阶段、消息或错误。迁移必须可重复检测，不重复添加列、事件或设置。

## 5. 写入与清理

任务创建时持久化 pending 快照和 queued 事件。任务获得实例锁并开始执行时，设置 started_at 并追加 started 事件。每次 Reporter.Progress 同时更新任务快照并追加 progress 事件。任务结束时设置终态和 finished_at，追加对应终态事件；失败事件消息使用实际错误原因。

任务快照更新和事件追加在同一个 SQLite 事务内完成，避免列表状态与展开日志不一致。任务操作的业务串行锁保持现状，不因事件持久化改变。

成功任务清理按全局 finished_at 降序决定保留集合；相同结束时间使用稳定主键作为次排序。每次任务成功后清理超出当前额度的最旧成功任务。管理员调低设置并保存成功后立即执行相同清理。任务删除通过外键级联删除事件。

系统设置更新与由该更新触发的清理在同一个事务内提交；任一步失败都不改变设置或删除任务。任务成功后的结果记录先独立提交，再尝试清理，避免清理故障覆盖真实任务结果。

pending、running、failed 和 interrupted 任务永不参与成功任务清理。调高上限不会恢复已经删除的任务。

## 6. HTTP 契约

保留 GET /api/jobs 和 GET /api/jobs/events 的现有摘要字段与 SSE 事件名。摘要增加开始和结束时间字段，但不携带事件数组，避免 SSE 每秒重复传输全部日志。

GET /api/jobs/{id} 保留当前顶层任务字段，并增加有序 Events 数组。现有调用方可以继续读取 Status、Percent、Error 等字段并忽略新增数据。

新增经过现有会话认证的设置接口：

- GET /api/settings/jobs 返回 successful_job_limit。
- PUT /api/settings/jobs 接受 successful_job_limit，仅允许整数 1 至 500。

设置请求格式错误或越界时返回 422，且不更新设置、不执行清理。设置持久化或清理失败时返回 500，界面保留原显示值并提示错误。

最近任务摘要接口继续遵守现有响应上限，避免未清理的失败任务让 SSE 响应无限增长。本次不新增历史分页。

## 7. 最近任务界面

采用行内手风琴交互。同一时间只展开一个任务，任务行使用可聚焦按钮语义并提供 aria-expanded 和关联面板标识。再次点击当前任务时收起。

摘要行展示：

- 任务类型和短 ID。
- 当前或最终阶段、消息摘要。
- 发起时间。
- 实际执行用时；尚未开始时显示排队中，运行中持续计算到当前时间。
- 进度和本地化状态。

展开时按需读取任务详情。已结束任务的日志在当前页面会话中缓存；运行中任务在 SSE 摘要的 UpdatedAt 变化后刷新详情。切换任务不会丢失已缓存的终态日志。

展开区顶部显示开始时间、结束时间、排队耗时和实际执行用时。下方按时间顺序显示事件类型、阶段、进度和消息。失败或中断原因置于顶部摘要，并在对应终态事件上使用现有错误色突出显示。

日志加载中使用稳定高度的轻量状态，失败时只在当前展开区显示错误和重试按钮，不隐藏任务列表。日志为空时显示明确空状态。

桌面端保持紧凑多列摘要；窄屏按信息优先级换行，展开日志使用横向不溢出的时间与内容两列布局。展开、加载或最长错误文本不能改变其他任务行的控件尺寸或造成内容重叠。

## 8. 系统设置界面

在现有系统设置页面增加独立的“任务记录”表单区。使用数字输入控件，最小值 1、最大值 500、默认显示服务端返回值，并提供明确的保存命令。

保存期间禁用重复提交。成功后保留服务端确认值；失败时使用现有设置错误区域展示原因。帮助文字说明该值只限制成功任务，失败和中断任务不会被自动删除。

## 9. 故障与恢复

- Panel 重启时，现有恢复流程继续把 pending 和 running 任务标记为 interrupted，同时补写中断结束时间和中断事件。
- 任务详情不存在时返回 404，展开区提示记录已被清理并允许收起。
- 单个任务详情加载失败不关闭 SSE，也不影响其他任务展开。
- SSE 断开时保留已有摘要和展开日志，并继续使用现有浏览器重连提示。
- 成功任务清理失败不得回滚已经正确记录的任务结果；接口或后台日志应保留清理失败证据，后续成功完成或设置保存会再次触发清理。
- 结构化事件消息沿用当前任务 Reporter 和错误文本，不额外包含凭据、请求头或原始命令输出。

## 10. 测试策略

- Store 迁移测试：新旧数据库升级、时间回填、合成事件、幂等迁移和级联删除。
- Store 行为测试：快照与事件事务写入、事件稳定顺序、默认设置、1 至 500 校验边界和成功任务精确清理。
- Job Manager 测试：排队、开始、多个进度事件、成功、失败、中断、开始/结束时间及成功后清理。
- HTTP 测试：摘要不携带日志、详情保留旧字段并返回事件、设置读取/保存、422 与 500 路径、降低上限后立即清理。
- React 测试：展开与收起、一次只展开一个、按需加载与缓存、运行任务刷新、失败原因、时间与时长格式、加载/错误/空状态和设置提交。
- E2E 测试：打开失败任务查看原因与事件，查看成功任务执行用时，在系统设置修改上限并确认页面保留规则。
- 回归验证：全部 Go 测试、go vet、前端单测、生产构建和 Playwright 控制面板流程。

## 11. 兼容边界

现有任务 ID、状态、进度、阶段、消息、错误、创建和更新时间字段保持不变。现有任务启动、按实例串行执行、轮询 GET /api/jobs/{id}、任务 SSE、计划任务和审计事件行为保持兼容。

新增字段和事件数组均为加法式扩展。旧任务可展开但只能显示迁移生成的最终快照。成功记录上限是唯一新增删除行为，并且只作用于成功终态。

## 12. 设计输入与影响

### TaskIntentDraft

让管理员从最近任务列表直接展开每个任务的结构化日志，尤其能定位失败原因；补充发起时间、排队耗时和实际执行用时；全局成功任务默认只保留最近 25 个，并可在系统设置中调整。

### BaselineReadSetHint

现有任务管理器只把最后一次 Reporter 进度覆盖写入 jobs，任务列表通过 SQLite 摘要和每秒 SSE 展示，详情接口返回同一任务快照。系统设置目前只有加密 Steam 与 GitHub 凭据接口，没有普通设置存储。Store 已使用版本化 SQLite 迁移并启用外键。

### ImpactStatementDraft

变更影响任务领域模型、Manager 与 Store 持久化接口、SQLite 迁移、任务 HTTP/SSE 契约、最近任务 React 页面和系统设置页面。兼容边界是保留现有任务执行与轮询行为；日志只记录 Panel 已知的结构化生命周期事件，不接管外部进程原始日志。
